# feat: Homarr path-only card URLs for multi-hostname access

## Overview

Switch Homarr-card hrefs generated by `homarr-container-adapter` from absolute (`https://<host>.local/cockpit/`) to path-only (`/cockpit/`), so they resolve against whatever hostname the user is currently on (mDNS, VPN FQDN, DHCP-DNS bare/short, DHCP domain). This change spans three repos: a patch to the `hatlabs/homarr` fork (relaxed schema + request-aware server-side resolver + bookmarks widget update), the adapter (validation, ping-URL builder, Signal K discovery, migration of existing stored cards), and the cockpit registry data file. An upstream PR to `homarr-labs/homarr` carries the fork patch back to the project; HaLOS-side work ships independently of the upstream outcome via an adapter-side migration that reconciles existing cards.

## Problem Frame

`halos-core-containers` PRs #122 / #123 made HaLOS devices answer correctly on multiple hostnames (TLS SANs, Authelia per-host cookies, OIDC redirect URIs all adapt per-request). Homarr cards do not — they bake `https://<host>.local/<path>` at registration time. A user who reaches the dashboard via VPN FQDN or DHCP-DNS hostname clicks a card and is jumped back to `.local`, which doesn't resolve in their context. This defeats the motivating use case of #122.

The chosen fix (Option D in the origin document) stores path-only hrefs and lets the browser resolve them against the current origin natively. Server-side consumers that need an absolute URL go through a request-aware resolver that uses `extractBaseUrlFromHeaders` (a primitive Homarr already uses for OIDC redirects). HaLOS-shipped cards always carry an explicit `pingUrl`, so the resolver's path-only fallback is rarely hit on HaLOS — the path-only-fallback branch matters mainly for upstream Homarr users who hand-edit apps without setting pingUrl.

## Scope Boundaries

- Out of scope: ping URL / health-check logic beyond what R2 (resolver) and R8 (`derive_ping_url`) require.
- Out of scope: changes to Authelia / OIDC / Traefik configuration.
- Out of scope: Homarr's integration framework internals.
- Out of scope: cross-origin SSO across hostnames.
- Out of scope: converting Signal K icon URLs to path-only.
- Out of scope: protocol-relative (`//host/...`) hrefs — explicitly rejected.
- Out of scope: a `/usr/share/halos/webapps.d/` convention for non-conffile package defaults.
- Out of scope: changes to integrations consuming `app.href` server-side beyond null-tolerance.
- **Out of scope: IP entries and single-label hostnames in the migration's self-hostname matching.** `lib-hostnames.sh` accepts these forms for cert-SAN purposes, but Authelia SSO does not work against IP literals or single-label hosts (RFC 6265 cookie-domain restriction; Authelia 4.39 rejects single-label cookie domains). Cards reachable only via IP or single-label hostname are not part of the supported multi-hostname access path. Stored cards using these forms remain untouched by the migration.
- **Out of scope: admin-created cards (cards not matching a registry entry).** R11/Unit 6 only migrates cards whose href matches a path-only registry entry shipped by HaLOS. Custom cards added by admins (third-party apps, custom dashboards, manual InfluxDB UIs) keep whatever href the admin entered. Admin-facing documentation should note that custom cards intended for multi-hostname access must be entered as path-only manually. Not trying to outsmart the admin.
- **Out of scope: deployments where an outer reverse proxy strips a path prefix** (e.g., admin fronts the device with their own proxy at `https://outer.example.com/halos/*` that strips `/halos/`). Path-only hrefs assume the device's Traefik is the origin's authoritative path-router. HaLOS does not support prefix-stripping outer proxies; admins who use them must continue to use absolute hrefs (and accept the original multi-hostname limitation).

## Key Technical Decisions

- **Path-only hrefs (Option D), not sentinel-host token (Option C).** Extends standard web link semantics; reuses Homarr's existing per-request hostname awareness. Carried from origin doc.
- **`resolveServerUrl(app, headers?)` is the centralization point for the fallback.** Backward-compatible: behavior for absolute hrefs is byte-identical; path-only adds a request-context expansion branch and a `null` fallback. Existing call sites all migrate to read through this helper. (Earlier draft included an env-configured `HOMARR_INTERNAL_BASE` branch — dropped as YAGNI; no current consumer.)
- **HaLOS-shipped cards always carry explicit `pingUrl`.** The path-only-fallback branch in `resolveServerUrl` is therefore upstream-correctness coverage, not a load-bearing path on HaLOS. This is what makes R8 a small, contained adapter-internal refactor of `derive_ping_url`.
- **Engage upstream PR-first.** No exploratory issue. Patch lands in `hatlabs/homarr` first, baked into the HaLOS Homarr container image; upstream PR opened from the same branch under the External Communication Policy.
- **Adapter-side migration (Unit 6) is the load-bearing ship guarantee, not the registry update (Unit 5).** Existing devices' stored cards get reconciled regardless of conffile-prompt outcome on `cockpit.toml`.
- **Self-hostname source for migration: `/etc/halos/hostnames.conf` directly** (the public contract from PR #122). Adapter applies the same `${hostname}` / `${fqdn}` / `${domain}` token-expansion as `lib-hostnames.sh`. Pre-#122 fallback: `gethostname() + .local`.
- **Migration match rule is tuple-tight:** rewrite only when (host matches an expanded entry, case-insensitive) AND (path equals the registry path) AND (scheme is `https` with no explicit port) AND (`pingUrl` is null or matches the adapter's previously-derived value). Anything else is treated as admin override and left untouched.
- **Sequencing requires Homarr fork patch in the deployed container before adapter starts emitting path-only.** Without the schema relaxation, the adapter's path-only push would be rejected. See Phased Delivery.

## High-Level Technical Design

> *This illustrates the intended approach and is directional guidance for review, not implementation specification. The implementing agent should treat it as context, not code to reproduce.*

### `resolveServerUrl(app, headers?)` resolution order (R2)

```
input: app = { href: string|null, pingUrl: string|null, ... }
       headers: ReadonlyHeaders | null  (optional)

returns: absolute URL string | null

step 1: if app.pingUrl is set
          -> return app.pingUrl                           [byte-identical to today]

step 2: else if app.href is absolute (parses as URL)
          -> return app.href                              [byte-identical to today]

step 3: else if app.href starts with "/" AND headers != null
          -> base = extractBaseUrlFromHeaders(headers)
          -> return base + app.href                       [new: per-request expansion]

step 4: return null                                       [new: caller handles null]
```

Caller behaviors:
- **Ping widget** (`packages/api/src/router/widgets/app.ts`): receives `null` → skip health check (don't render an indicator, or render "unknown").
- **Cached integration job handler** (`packages/request-handler/src/lib/cached-request-integration-job-handler.ts`): receives `null` → `externalUrl: null` (already the existing pattern, just becomes reachable for path-only).

### Adapter migration matching rule (R11/Unit 6)

```
For each stored Homarr card with absolute href:
  parse href into (scheme, host, port, path)

  if scheme == "https" AND port is default (443) AND host ∈ self_hostnames AND path matches a path-only registry entry:
    -> rewrite stored href to the path-only form
  else:
    -> leave untouched (admin override)

self_hostnames =
  if /etc/halos/hostnames.conf exists:
    expand each line using ${hostname} / ${fqdn} / ${domain} semantics
  else:
    [gethostname() + ".local"]    -- pre-#122 fallback
```

- **Error propagation:** `resolveServerUrl` returns `null` for the new path-only-without-context case. Callers must handle null. Existing absolute-href + pingUrl behavior is byte-identical, so error paths for the historic shape don't change.
- **State lifecycle risks:** The migration in Unit 6 mutates stored Homarr cards in-place via `app.update`. If the migration runs twice (transiently — e.g., adapter restart mid-sync), the second run is a no-op because the stored href already matches the registry path-only form. If the migration is interrupted partway, only some cards are rewritten; subsequent sync completes the rest.
- **API surface parity:** `appHrefSchema` is exported and consumed by tRPC mutations (`app.create`, `app.update`, etc.). The relaxation is additive; existing absolute-form callers are unaffected. Path-only callers gain access; the resolver shields server-side consumers from the new shape.
- **Integration coverage:** Path-only hrefs interact with Authelia (browser session cookie attaches to user-initiated requests; background-job pings without a session cookie hit Authelia's login redirect — same as today's absolute-href behavior). The pingUrl widget's tolerance of the redirect chain is empirically verified.
- **Unchanged invariants:** Authelia / OIDC / Traefik configuration. Existing absolute-href behavior at every Homarr call site. Today's pingUrl mechanism for HaLOS-shipped cards (the new path-only branch in `derive_ping_url` produces a URL identical in shape to today's absolute-href output). The bookmarks widget's hostname rendering for absolute hrefs.

## Risks & Dependencies

| Risk | Mitigation |
|------|------------|
| **Sequencing violation between Homarr container deployment and adapter package upgrade** (apt and docker-image lifecycles are decoupled — adapter package can upgrade before docker-compose pull refreshes Homarr). | Unit 6's migration runs a Homarr-version probe (test-create a path-only app); if Homarr rejects it, migration logs "deferred" and exits; next sync retries. Migration is safe to deploy independently of docker timing. (The originally-planned APT `Depends: homarr-container-adapter (>= <version>)` pin in `halos-cockpit-config` was dropped as redundant — see Unit 5.) |
| **R5 audit surfaces deeply-coupled consumer that can't migrate cleanly.** | Per origin doc: escalate as a blocker before submitting upstream PR. Unit 1 deliverable explicitly includes the audit results in the PR description. Audit scope explicitly extended to packages/validation, packages/auth, packages/db, and import/export flows. |
| **Hostname-list edge cases (`${fqdn}` resolution failure, hostname-rename history) cause migration to mis-classify cards.** | Unit 6 unit tests cover the edge cases. Match rule (scheme + default-port + host + path + pingUrl-not-admin-edited) errs toward "leave untouched" — a missed migration produces a duplicate (visible, recoverable manually) rather than a silent overwrite of an admin's intentional edit. |
| **Admin pingUrl override silently overwritten by migration.** | Unit 6 extends `SelectableApp` to deserialize `pingUrl`; match rule preserves both `href` AND `pingUrl` when stored `pingUrl` differs from the adapter-derived value. Explicit test scenario covers the case. |
| **Multi-client ping subscription breaks under path-only hrefs in upstream Homarr** (different clients on different hostnames compute different filter keys). | Unit 1 changes the redis subscription channel key from `pingUrl ?? href` to `app.id`-based. App IDs are stable across clients. Documented in upstream PR as a small correctness improvement. |
| **Upstream PR rejected or stalls.** | Origin doc names the pivot: keep carrying the patch in `hatlabs/homarr` as a permanent diff; consider Option C as an alternative PR. Re-evaluation date: 2026-10-01. HaLOS user outcomes are unaffected; Unit 7 is community-contribution work, not on the HaLOS critical path. |
| **Existing pingUrl mechanism (`https://host.docker.internal<path>`) is more fragile than the empirical probe suggests** (e.g., works under specific Authelia/Cockpit conditions but breaks for new backends). | Today's behavior is verified as working on `halosdev.local`. Unit 3's path-only branch produces byte-identical URL shape to absolute branch. Any fragility that surfaces affects today's behavior equally and is out of scope — file a separate issue. |
| **Bookmarks widget path-rendering is awkward for nested paths** (e.g., `/signalk-server/@signalk/freeboard-sk/` renders as `/signalk-server/@signalk/freeboard-sk` — long string). | Acceptable. Path is the disambiguator regardless of length. Truncation/styling refinement deferred as a follow-up if reviewers raise concerns. |
| **External Communication Policy violation:** an autonomous tool action against `homarr-labs/homarr`. | Unit 7 explicitly gates all external actions. Plan-level reminder. The implementing agent MUST present-and-wait, not submit-and-tell. |

## Documentation / Operational Notes

- The path-only-vs-absolute *convention* belongs in user-facing docs (docs.halos.fi), not in `halos-cockpit-config/AGENTS.md`. The cockpit.toml inline comment covers the editing-site question.
- **`homarr-container-adapter/AGENTS.md`** updated (or its README) to document: the path-only/absolute validation in webapps.d, the migration's read of `/etc/halos/hostnames.conf`, the `is_self_origin` predicate's match rule, the admin-pingUrl-preservation rule, and the version probe.
- **Admin-facing documentation** (e.g., HaLOS docs.halos.fi or Homarr admin guide): note that custom admin-created cards are NOT migrated automatically. Admins who want their custom cards to work across multiple hostnames must enter them as path-only manually (e.g., `/my-custom-app/` instead of `https://halos.local/my-custom-app/`). HaLOS-shipped cards are handled by the adapter migration; admin overrides are respected entirely.
- **Operational rollout:** standard apt-package upgrade flow on devices. The adapter's first post-upgrade sync runs the migration. No admin action required for mDNS-only devices (default case). Admins on multi-hostname setups continue to manage `/etc/halos/hostnames.conf` per #122/#123 — no changes there.
- **Monitoring:** journalctl on the adapter for hostname-load warnings (malformed `hostnames.conf` lines, `${fqdn}` resolution failure) and for migration-deferred messages (Homarr container hasn't yet been refreshed with the patched fork — the migration logs and exits, retries next sync). The adapter emits its own log strings (no requirement to match `lib-hostnames.sh`'s diagnostic signatures).
- **Rollback:** if the adapter migration produces unexpected card duplication or loss, an admin can revert the adapter package to the previous version and re-sync; stored Homarr cards in either form (absolute or path-only) remain readable by Homarr regardless of which adapter version processes them next.
